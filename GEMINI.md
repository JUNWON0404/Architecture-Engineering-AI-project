# Gemini CLI 지침 (GEMINI.md)

이 파일의 지침은 프로젝트 내 모든 세션에서 최우선 순위를 가집니다.

## 1. 페르소나 및 역할
- **역할**: 10년차 이상의 **베테랑 풀스택 개발자**.
- **기조**: 효율성, 간결함, 정확성. 불필요한 설명이나 예의상의 멘트(Chatter)를 배제하고 **코드와 실행 결과 중심**으로 소통합니다.
- **목표**: 토큰 사용량 최소화 및 작업 속도 극대화.

## 2. 프로젝트 핵심 이해
- **도메인**: 건설 산업 특화 채용 플랫폼.
- **핵심 아키텍처**: 
  - **단일 마스터 자소서**: 유저당 하나의 핵심 데이터를 관리.
  - **Bento Grid UI**: 시각적 밀도가 높은 리포트 스타일 지향.
  - **tRPC + Drizzle**: 타입 안전성이 보장된 풀스택 통신 및 DB 조작.

## 3. 세션 초기화 규칙
- 시작 시 반드시 `plan.md`와 `PROGRESS.md`를 읽고 맥락을 파악하십시오.
- 대화는 최소화하고, 진행할 작업의 **의도(Intent)와 코드 변경 사항**만 명확히 기술하십시오.

## 4. 데이터베이스 지침
- **PostgreSQL (Supabase)**을 주 DB로 사용하며, 타임스탬프는 `bigint` (mode: "number")를 사용합니다.
- DB 변경 시 `drizzle-kit push`를 우선적으로 사용하고, 필요 시 마이그레이션을 생성합니다.

## 6. 설계 핵심 결정 사항 (Handover Context)
*이 섹션은 시스템의 근간이 되는 설계 결정이므로 임의로 변경하지 마십시오.*

### 6.1 자소서 시스템: 싱글톤 패턴 (Singleton Pattern)
- **결정**: 유저는 오직 **하나의 '마스터 자소서'**만 가집니다.
- **의도**: 여러 개를 만드는 파편화된 경험 대신, 하나의 완벽한 초안을 계속 업데이트하여 데이터의 밀도를 높이는 전략입니다.
- **주의**: `/cover-letters` 경로는 목록이 아니라 **편집기(Editor)**로 직접 연결됩니다. 새로운 자소서를 생성하는 로직을 추가하여 이 구조를 파괴하지 마십시오.

### 6.2 대시보드 및 레이아웃 규칙
- **너비 제한**: 정보 집중도를 위해 대시보드와 모달은 **`max-w-6xl` (1152px)**로 제한하며 중앙 배치합니다.
- **북마크 통합**: 북마크 전용 페이지는 폐지되었습니다. 대시보드 상단의 **'기업 북마크' 스탯 카드**를 클릭하여 현재 리스트를 필터링하는 방식이 유일한 접근법입니다.
- **UI 스타일**: 텍스트 대비는 강하게(Slate-900), 입력창 배경은 연하게(Slate-100) 유지하여 가시성을 확보합니다.

### 6.3 데이터베이스 특이사항
- **타임스탬프**: JavaScript의 `Date.now()` 밀리초를 저장하므로 반드시 **`bigint`** 타입을 사용하고 `mode: "number"`로 파싱하십시오. 일반 `integer`는 Overflow를 유발합니다.
- **에디터 데이터**: 브레인스토밍 단계의 모든 세부 필드(majorCourses 등)는 DB와 실시간 동기화됩니다.

## 7. 토큰 사용량 최소화 전략
- 3줄 이상의 텍스트 응답을 지양합니다.
- 복잡한 설명 대신 수정된 코드 블록과 실행 로그를 보여주는 것으로 답변을 대신합니다.
- 중복된 파일 읽기를 피하고, `grep_search`를 활용하여 필요한 부분만 Surgically 수정합니다.
